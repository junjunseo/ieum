# 이음 (Ieum)

> 한국어의 통사 구조를 반영한 프로그래밍 언어

**이음**은 단어와 조사를 잇고, 코드와 자연어를 잇는다는 의미입니다. 한국어가 교착어로서 조사를 통해 문법적 관계를 표현하는 특성에서 이름을 따왔습니다.

---

## 프로젝트 동기

기존 한국어 프로그래밍 언어들(씨앗 등)은 두 가지 한계가 있었습니다.

1. 구현체와 명세가 단절되어, 현재 학습하거나 계승할 수 있는 소스가 남아 있지 않습니다.
2. 상당수가 영어 언어의 문법에 한글 단어만 대응시킨 수준에 머물렀습니다.

이음은 단순한 단어 치환이나 UI 번역(기존 익스텐션으로 가능한 수준)을 넘어, **한국어의 어순(SOV)과 조사 구조를 파싱 단계에서 직접 다루는 것**을 목표로 합니다. 이는 기존 도구의 확장으로는 불가능한, 컴파일러 설계상의 비자명한 기술적 문제입니다.

---

## 학습 목표

이 프로젝트는 시장에 출시할 제품이 아니라, **언어 처리 시스템을 직접 설계·구현하며 컴파일러 이론을 체득하는 것**을 일차 목표로 합니다.

- 렉싱 · 파싱 · AST 구성 · 평가에 이르는 언어 처리 파이프라인 전체를 직접 구현한다.
- 기존 언어의 문법을 그대로 차용하지 않고, 한국어라는 제약 조건 위에서 직접 설계 결정을 내린다.
- 트리워킹 인터프리터 대 바이트코드 VM, 타입 시스템 도입 여부, 에러 복구 전략 등 주요 설계 트레이드오프를 문서화한다.

---

## 설계 목표

- 한국어 통사 구조(어순 · 조사)를 반영한 문법 설계
- 렉서 → 파서 → AST → 인터프리터 전 과정의 직접 구현
- 단계적 접근: **1단계** 조사 기반 키워드 문법(어순 고정) → **2단계** SOV 어순 유연화(도전 과제)

---

## 아키텍처

```
소스 코드 → Lexer → Parser → AST → Interpreter → 실행
```

---

## 현재 진행 상황

이 문서는 이음의 장기 언어 아이디어를 설명한다. 현재 1학기 구현 범위는 범용 한국어 인터프리터가 아니라, **구조적으로 잘못된 모듈 의존 관계를 실행 전에 거부하는 구조 검사 코어**에 맞추어 축소했다.

- [x] Lexer — `module`, `depends`, `layer`, `above` 기반 구조 선언 토큰화
- [x] Parser — 모듈·의존·계층 선언을 AST로 변환
- [x] Checker — 선언 유효성, 암묵적 의존, 순환 의존, 계층 위반 검사
- [x] CLI — `.ieum` 파일 파싱 결과와 구조 검사 결과 출력
- [x] 테스트 — 파서, Lexer → Parser → Checker 통합, 검사기 시나리오 자동화
- [ ] Interpreter — 모듈 내부 로직 평가 및 실행 (2학기 이후 확장)
- [ ] SOV 어순 확장 — 한국어 어순과 조사 기반 문법 실험 (장기 도전 과제)

## 예제 (목표 문법)

```
점수 = 95
만약 점수 >= 90 이면:
    출력("잘했어요!")
아니면:
    출력("다시 해봐요")

반복 3 번:
    출력("안녕하세요")

횟수 = 0
반복 횟수 < 5 동안:
    출력(횟수)
    횟수 = 횟수 + 1
```
