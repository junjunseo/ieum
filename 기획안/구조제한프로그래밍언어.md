# 종합 설계 프로젝트 기획서

**프로젝트명(가제):** 구조적 코드 품질을 언어 차원에서 강제하는 프로그래밍 언어
**작성자:** 임준서
**구분:** 개인 졸업 종합설계 프로젝트 (1년 / 졸업 전시 포함)

---

## 1. 문제 정의 (Problem Definition)

### 배경
생성형 AI를 활용한 이른바 "바이브코딩"이 확산되면서, 동작은 하지만 구조적으로 열악한 코드가 빠르게 양산되고 있다. 이러한 코드는 의존 관계가 무분별하게 얽히고, 모듈 간 경계가 무너지며, 계층 구조가 지켜지지 않는 경향이 있다. 그 결과 단기적으로는 작동하더라도 규모가 커질수록 유지보수성·확장성이 급격히 저하된다.

소프트웨어 공학의 관점에서 코드의 가치는 작성(coding) 그 자체보다 전체 구조의 품질에 의해 결정된다. 그러나 파이썬·자바스크립트 등 널리 쓰이는 범용 언어는 구조적으로 나쁜 코드를 *너무 쉽게 허용*한다. 한 파일에 모든 책임을 몰아넣거나, 임의의 모듈을 아무 곳에서나 호출하거나, 순환 의존을 만들어도 언어는 이를 막지 않는다.

### 핵심 문제
**범용 언어는 "구조적으로 나쁜 코드"를 작성하는 것을 구조적으로 방지하지 못한다.** 좋은 구조는 전적으로 작성자의 규율에 의존하며, AI가 코드를 대량 생성하는 환경에서 이 규율은 더욱 지켜지기 어렵다.

### 문제의 구체화 — 다루는 범위
"나쁜 코드"는 광범위하므로, 본 프로젝트는 객관적으로 검출 가능한 **의존성 및 모듈 구조의 결함**으로 범위를 한정한다.

- 암묵적 의존: 모듈이 선언하지 않은 다른 모듈을 임의로 사용
- 순환 의존: 모듈 A → B → A 형태의 상호 참조
- 계층 위반: 하위 계층(예: 저장소)이 상위 계층(예: UI, 서비스)을 역방향으로 호출

## 2. 해결 방안 (Solution)

### 접근의 핵심
위 결함들을 **컴파일 타임에 거부**하는 작은 프로그래밍 언어와 그 컴파일러를 직접 설계·구현한다. 구조적으로 나쁜 코드는 *애초에 컴파일되지 않는다*. 좋은 구조를 권장하는 것이 아니라, 나쁜 구조를 언어 차원에서 강제로 차단하는 것이 본 언어의 정체성이다.

### 언어가 강제하는 규칙 (1학기 목표)
1. **명시적 의존 선언** — 모듈은 자신이 사용하는 다른 모듈을 반드시 선언해야 하며, 선언하지 않은 모듈을 호출하면 컴파일 오류
2. **순환 의존 금지** — 모듈 간 의존 그래프에서 사이클이 검출되면 컴파일 오류
3. **계층 위반 금지** — 사전에 정의된 계층 순서를 직접 또는 전이적으로 위반하는 호출은 컴파일 오류

### 동작 개념 (예시 스케치)
```
계층 [UI > 서비스 > 저장소]

모듈 주문 (계층: 서비스) {
    사용: 결제, 재고          // 선언한 모듈만 호출 가능
    공개: 주문하기()
}

모듈 결제 (계층: 서비스) {
    사용: 저장소
    공개: 결제처리()
}
```
- `주문`이 선언하지 않은 `알림`을 호출 → **컴파일 오류 (암묵적 의존)**
- `결제`가 다시 `주문`을 사용하도록 선언 → **컴파일 오류 (순환 의존)**
- 저장소 계층 모듈이 `주문` 같은 상위 계층 모듈을 사용 → **컴파일 오류 (계층 위반)**

*위 문법은 개념 설명용 스케치이며 최종 문법은 구현 과정에서 확정한다.*

## 3. 기존 기술과의 차별성

| 구분 | 기존 방식 | 본 프로젝트 |
|------|-----------|-------------|
| 명시적 의존 선언 | Java 모듈 시스템(`requires`/`exports`)이 일부 지원, 단 선택적 | 언어에 내장, 강제, 비활성화 불가 |
| 순환 의존 | Java는 패키지·클래스 수준에서 허용(막지 않음) | 컴파일 타임 거부 |
| 계층 위반 | 언어가 아닌 외부 테스트 도구(예: ArchUnit)로만 검사 | 언어 차원에서 검사 |
| 의존성 시각화 도구 | 이미 작성된 코드를 *사후 분석*하여 추정 | 언어가 구조를 보장하므로 *사전 예방* + 정확한 정보 |

### 차별점 요약
기존 언어와 도구는 이러한 규칙들을 **(1) 기능마다 분산된 형태로, (2) 일부는 언어 외부 도구로, (3) 선택적으로** 지원한다. 본 프로젝트는 이를 **하나의 일관된 언어에 내장하여 강제**한다. "할 수 있다"와 "강제되어 어길 수 없다"는 구조적으로 다르며, 후자가 AI 코드 양산 환경에서 갖는 의미가 본 프로젝트의 핵심 기여이다.

또한 기존 의존성 시각화·분석 도구가 "이미 짜인 나쁜 코드"를 사후 진단하는 것과 달리, 본 언어는 나쁜 코드를 처음부터 작성 불가능하게 만든다(사후 진단 vs 사전 예방).

## 4. 기대 성과 및 가치 (Outcome Value)

- 정의된 구조적 결함(암묵적 의존·순환 의존·계층 위반)을 컴파일 타임에 차단함을 **테스트 케이스로 정량 증명** — "올바른 코드는 통과, 결함 코드는 N종류 모두 거부"
- 나쁜 구조의 예시 코드를 본 언어로 작성했을 때 컴파일러가 이를 거부하는 과정을 **졸업 전시에서 시연**
- 컴파일러 프론트엔드(어휘 분석 → 구문 분석 → 의존 그래프 구성 → 규칙 검사)를 직접 구현함으로써, 정적 분석 기반 품질 보장 메커니즘의 구현 역량을 입증

## 5. 구현 계획 (Implementation Plan)

### 1학기 — 구조 검사 코어
| 단계 | 내용 |
|------|------|
| 어휘 분석기(Lexer) | 토큰화 |
| 구문 분석기(Parser) | 모듈·의존·계층 선언 구문 → AST |
| 의존 그래프 구성 | AST로부터 모듈 간 의존 관계 그래프 추출 |
| 규칙 검사기 | 암묵적 의존·순환 의존(사이클 검출)·계층 위반 검사 |
| 검증 | 테스트 케이스 5~10개 (정상/결함 코드) |

### 현재 구현 상태

- Lexer, Parser, AST, Checker, CLI를 통해 구조 선언 파일을 검사하는 1학기 코어를 구현했다.
- 검사기는 선언 유효성, 암묵적 의존, 순환 의존, 직접·전이적 계층 위반을 모두 실행 전에 거부한다.
- 자동 테스트는 파서, 전체 파이프라인, 구조 검사기 시나리오를 포함한다.
- 시연은 정상 구조 1개와 결함 구조 예제들을 실행하여 "통과해야 하는 코드는 통과하고, 결함 코드는 거부됨"을 보여주는 방식으로 진행한다.

### 2학기 — 확장 (로드맵)
- 모듈 내부 로직(변수·함수·제어 흐름) 문법 및 인터프리터/컴파일러 백엔드 추가
- 의존 그래프 및 변경 영향 범위 시각화 (전시용)

### 범위 관리 원칙
1학기의 목표는 "완성"이 아니라 **작동하는 코어 + 명확한 문제 정의의 입증**이다. 1년 분량을 무리하게 1학기에 압축하지 않고, 검증 가능한 최소 핵심을 먼저 완성한 뒤 단계적으로 확장한다.
